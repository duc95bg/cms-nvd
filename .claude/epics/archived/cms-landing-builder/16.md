# Bug: indexed-list JSON string not decoded on site update

## Context
Found while manually testing issue #11 (`SiteController`): after creating a seeded product site and visiting `/admin/sites/{id}/preview`, the page throws:

```
ErrorException
foreach() argument must be of type array|object, string given
at resources/views/templates/product.blade.php:25
@foreach (data_get($site->content, 'features.items', []) as $feature)
```

## Description
The edit form (from #13) renders indexed-list nodes (e.g. `features.items`) as a single `<textarea>` containing `json_encode($value, ...)`. The user edits and submits. `SiteController::update()` (from #11) rehydrates the form payload with `data_set($nested, $dotKey, $value)` but **never decodes the JSON string back into an array**. The string lands in `sites.content` and the next render of any template that loops over that node explodes.

The round-trip create → edit → save → preview is broken for any template whose `default_content` contains an indexed list — which is every non-trivial template.

## Steps to Reproduce
1. `php artisan migrate:fresh --seed` (seeds the `product` template)
2. Register a user via `/register`, then POST `/admin/sites` with `template_id=1`, `slug=demo`
3. Go to `/admin/sites/1/edit` (form loads correctly)
4. Click Save without changing anything
5. Click Preview → 500 error

Alternatively the feature test `test_admin_update_rehydrates_dot_notation_into_nested_json` passes because it only submits scalar/translatable fields — it does not exercise the list round-trip. That gap is why #11 shipped green.

## Expected vs Actual
- **Expected:** After save + preview, the product template renders `features.items` as 3 feature cards.
- **Actual:** 500 because `$site->content['features']['items']` is a JSON string, not an array.

## Acceptance Criteria
- [ ] `SiteController::update()` decodes JSON strings that parse into arrays before calling `data_set`, so indexed lists survive the round trip.
- [ ] Strings that don't parse as JSON stay as strings (no silent data loss for legitimate string values that happen to start with `{` or `[`).
- [ ] A new feature test `test_admin_update_preserves_indexed_list_round_trip` in `SiteCmsTest`:
  - Creates a site from the seeded template
  - Performs a `PUT` that submits `content[features.items]` as the **JSON string** of the original items (simulating what the edit form actually sends)
  - Asserts the saved `$site->fresh()->content['features']['items']` is an array with 3 entries, same structure as the seed
- [ ] The preview route for that site returns 200 and `assertSee` the first feature title
- [ ] Existing 13 tests still pass (`SiteCmsTest` 9/9 + `SiteContentFlattenTest` 4/4)
- [ ] Original issue #11 behaviour for scalar/translatable fields is unchanged
