# 39 — Feature tests

**Size:** L (3h)

## Description

Write comprehensive feature and unit tests for the cart-checkout epic. This includes a feature test class covering the full cart and checkout flow (14 tests) and a unit test class for PaymentService (4 tests). Tests use RefreshDatabase and seed with CatalogSeeder to have products and variants available.

This task depends on all previous tasks being complete.

## Acceptance Criteria

1. All 14 feature tests pass.
2. All 4 unit tests pass.
3. Tests cover the happy path and key error paths.
4. Tests verify stock management (decrement on checkout, validation on add).
5. Tests verify order data integrity (snapshots, status logs).
6. Tests verify authorization (user can only see own orders).
7. Tests verify admin status transitions (valid and invalid).
8. No existing tests are broken.

**Test 1: `test_add_to_cart_stores_item_in_session`**
- Get a product (with stock > 0).
- POST `/cart/add` with product_id, variant_id (null if no variants), qty=1.
- Assert redirect back with session 'success'.
- Assert session('cart') contains the item keyed correctly.
- GET `/cart` and assert the product name appears in the response.

**Test 2: `test_add_to_cart_validates_stock`**
- Get a product/variant with known stock (e.g., stock=2).
- POST `/cart/add` with qty=999.
- Assert redirect back with session 'error' containing "Insufficient stock" or similar.
- Assert session('cart') does NOT contain the item (or is empty).

**Test 3: `test_add_to_cart_requires_variant_when_product_has_variants`**
- Get a product that has variants.
- POST `/cart/add` with product_id but variant_id=null.
- Assert validation error or appropriate error handling.
- (This test depends on whether the validation requires variant_id for products with variants — adjust based on implementation.)

**Test 4: `test_cart_update_quantity`**
- Add an item to cart (POST `/cart/add`).
- POST `/cart/update` with key and new qty.
- GET `/cart` and assert the updated quantity appears.

**Test 5: `test_cart_remove_item`**
- Add an item to cart.
- POST `/cart/remove` with key.
- GET `/cart` and assert empty cart state (or item not present).

**Test 6: `test_cart_page_shows_items_and_total`**
- Add 2 different items to cart.
- GET `/cart`.
- Assert both product names appear in response.
- Assert total is displayed (check for formatted amount).

**Test 7: `test_checkout_page_requires_non_empty_cart`**
- GET `/checkout` with empty cart.
- Assert redirect to cart index.
- Assert session has 'error'.

**Test 8: `test_checkout_cod_creates_order_and_decrements_stock`**
- Get product/variant, note initial stock.
- Add item to cart (qty=2).
- POST `/checkout` with valid customer data and payment_method=cod.
- Assert redirect to success route.
- Assert Order exists in DB with correct customer data, payment_method=cod, status=pending.
- Assert OrderItem exists with correct product snapshot and qty.
- Assert product/variant stock decreased by 2.
- Assert cart is cleared (session cart is empty).

**Test 9: `test_checkout_bank_transfer_shows_bank_info`**
- Add item to cart.
- POST `/checkout` with payment_method=bank_transfer.
- Follow redirect.
- Assert response contains bank name from config.
- Assert response contains the order number.

**Test 10: `test_order_items_snapshot_product_data`**
- Add item to cart, checkout with COD.
- Get the created order with items.
- Assert order item's `product_name` matches the product's name at time of order.
- Assert order item's `product_price` matches the product/variant price.
- Update the product's name and price in DB.
- Re-fetch the order item — assert it still has the OLD name and price (snapshot integrity).

**Test 11: `test_admin_order_list_shows_orders`**
- Create an order (via checkout or factory).
- Acting as admin user, GET `/admin/orders`.
- Assert 200 status.
- Assert response contains the order number.

**Test 12: `test_admin_order_status_update_with_log`**
- Create a pending order.
- Acting as admin, POST `/admin/orders/{order}/status` with status=confirmed.
- Assert order's status is now 'confirmed' in DB.
- Assert OrderStatusLog exists with old_status=pending, new_status=confirmed, changed_by=admin id.

**Test 13: `test_admin_order_status_invalid_transition_rejected`**
- Create a pending order.
- Acting as admin, POST `/admin/orders/{order}/status` with status=delivered (invalid: can't go from pending to delivered).
- Assert redirect back with session 'error'.
- Assert order's status is still 'pending' in DB.

**Test 14: `test_user_order_history_shows_own_orders_only`**
- Create 2 users: user A and user B.
- Create an order for user A and an order for user B.
- Acting as user A, GET `/orders`.
- Assert response contains user A's order number.
- Assert response does NOT contain user B's order number.
- Acting as user A, GET `/orders/{userBOrder}/detail`.
- Assert 403 status.

**Test 1: `test_vnpay_url_contains_required_params_and_hash`**
- Create an Order (use factory or manual creation).
- Call `PaymentService::createVnpayUrl($order)`.
- Parse the returned URL.
- Assert URL starts with the configured VNPay URL.
- Assert query contains: vnp_TmnCode, vnp_Amount, vnp_TxnRef (=order_number), vnp_ReturnUrl, vnp_SecureHash.
- Assert vnp_Amount equals `$order->total * 100`.

**Test 2: `test_vnpay_hash_verification_valid`**
- Set config values for vnpay hash_secret.
- Build a params array with known values.
- Compute the expected HMAC-SHA512 hash manually.
- Add vnp_SecureHash to params.
- Call `PaymentService::verifyVnpayHash($params)`.
- Assert returns true.

**Test 3: `test_vnpay_hash_verification_invalid`**
- Same as above but tamper with one param value after hashing.
- Call `PaymentService::verifyVnpayHash($params)`.
- Assert returns false.

**Test 4: `test_paypal_usd_conversion`**
- Set config `payment.paypal.usd_rate` to 25000.
- Create an order with total = 500000 (VND).
- Assert that the USD amount would be 20.00.
- (This may test a helper method or be verified indirectly through the PayPal order creation. If PaymentService has a public/testable conversion, test it directly. Otherwise, mock Http and verify the amount in the API call.)

### Test helpers and setup

**Admin user creation:**
```php

$this->actingAs($admin);
```

**Product/variant access:**
After seeding with CatalogSeeder, get products:
```php
$product = Product::whereHas('variants')->first();
$variant = $product->variants->first();
```

Or for a product without variants:
```php
$product = Product::doesntHave('variants')->first();
```

**Cart item key format:** `"{productId}-{variantId}"` or `"{productId}-"` for null variant.
